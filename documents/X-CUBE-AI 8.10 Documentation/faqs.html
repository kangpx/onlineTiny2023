General

How to know the version of the deep-learning framework components which are used?

X-CUBE-AI Expansion Package is a complete self-contained application package. No external tools is requested to use the package. To know the version of the main components which are embedded, the following command can be used:

>  stm32ai --tools_version
Neural Network Tools for STM32AI v1.5.1 (STM.ai v7.0.0)
- Python version   : 3.7.9
- Numpy version    : 1.19.5
- TF version       : 2.5.0
- TF Keras version : 2.5.0
- ONNX version     : 1.6.0
- ONNX RT version  : 1.7.0

Note — User should be aware that the respective Python DL modules are used to run the original model and also to import/parse the model. To know the supported layers/operators, please refer to the description in the [TFLITE], [KERAS] and [ONNX].

How is used the CMSIS-NN library?

The network runtime library is partially based on the CMSIS library. CMSIS is a vendor-independent hardware abstraction layer for micro-controllers that are based on Arm® Cortex® processors (https://arm-software.github.io/CMSIS_5/General/html/index.html). The CMSIS-NN sub-component and a minor part of the CMSIS-DSP sub-component are embedded in the network_runtime.a library to be sure that the requested files are compiled with the optimal options.

component version
CMSIS package r5.7.0
CMSIS-Core(M) V5.4.0
CMSIS-DSP V1.8.0
CMSIS-NN V1.3.0

A part of the forward kernel functions are directly mapped on the CMSIS-NN (legacy support of the Qmn format) when available but the major part, are the optimized implementations (integer and float format) only based on the CMSIS-type definitions for portability across the STM32 families.

What is the EABI used for the network_runtime libraries?

For performance motivations, the provided libraries (network_runtime.a) are compiled with the hard floating-point ABI option allowing generation of floating-point instructions and using FPU-specific calling conventions. This implies that the final end-user project should be also compiled and linked with this option.

Note — Abnormal situation is normally detected at link time, but there is a specific case where the firmware is generated w/o errors but at run-time the results are UNPREDICTABLE. For example for a GCC-base environment, -mfloat-abi=soft or softfp can be set by default for the whole project to use an other EABI-soft-base binary library. There is currently no link issue because the library has been generated with another EABI compliant tool-chain (IAR for ARM tool-chain).

If specific binary library version is requested see with your local ST support.

X-CUBE-AI Python API availability?

X-CUBE-AI is only available through an UI interface (fully integrated in STM32CubeMX) and through a command line interface.

TensorFlow Keras (tf.keras) vs Keras.io

X-CUBE-AI 6.0 pack embeds only TensorFlow 2.3.1. The Keras.io module is no more used. 'tf.keras 2.4.0' is now used to import and to validate the Keras models. The previous 'TF_KERAS=False' environment variable can be no more used to use a Kerias.io back-end. This implies that the Keras model with a NCHW tensor format (channel-first) can be no more validated. Code can be always generated but the validation flow to test the outputs against the outputs of the generated c-model can be no more used. Following type of error message will be generated during the execution of the provided model.

...
INTERNAL ERROR: The Conv2D op currently only supports the NHWC tensor format
                on the CPU. The op was given the format: NCHW [Op:Conv2D]
...

If the user wants to validate the generate c-model, he must provide a test data set. The '--no-exec-model' option can be used to avoid to execute the imported model.

> stm32ai validate -m <keras_model_with_NCHW>.h5 -vi test.npz --no-exec-model
...

Evaluation report (summary)
------------------------------------------------------------------------------------------------------
Mode              acc      rmse      mae       l2r       tensor
------------------------------------------------------------------------------------------------------
x86 C-model #1    100.00%  0.000000  0.000000  0.000000  activation_6 [ai_float, (1, 1, 2), m_id=20]

It is possible to update a model on the firmware w/o having to do a full firmware update?

Yes - The '--binary' and '--relocatable' options of the 'generate' command allow to implement a simple or complete mechanism to be able to upgrade a whole generated c-model w/o having to do a full firmware update (refer to [CLI] or [RELOC], "Relocatable binary model support" article).

Keras Model or Sequential layer support?

Nested topologies are not supported. This can appear when the Keras functional API is used to build the network and a 'Model' object is called as a layer.

A possible work-around is to convert the Keras model to a TensorFlow lite model, if all operators are supported.

converter = tf.lite.TFLiteConverter.from_keras_model_file(<keras_model_path>)
model = converter.convert()

Is it possible to split the weights buffer?

Yes, by weights/bias tensors, see the '--split-weights' option (refer to [CLI]).

Is it possible to place the “activations” buffer in different memory segments?

No, only a continuous memory-mapped buffer should be provided by the AI application (refer to [API]).

How to compress the non-dense/fully-connected layers?

Only the 32b floating-point dense/fully-connected layer can be compressed.

Is it possible to apply a compression factor different of x8, x4?

No. The underlying weight-sharing algorithm (K-means clustering) is based on a dictionary with 16 (x8) or 256 (x4) entries. At the end, the global gain depends of the parameter number versus dictionary size. Note that for a given layer, bias parameters are not necessarily compressed. Compression is applied by tensor.

How to specify or to indicate a compression factor by layer?

By default, the compression process tries to apply globally the same compression factor (x4 or x8) for all dense-layers or fully-connected-layers. If the global accuracy is too much impacted, the user has the possibility to refine the expected compression factor layer-by-layer.

A JSON file must be defined, to indicate what is the compression factor (8 or 4) which must be applied for a given layer. The layer is specified by its original name.

Example of configuration file:

{
    "layers": {
        "dense_1": {"factor": 8},
        "dense_2": {"factor": 4},
        "dense_3": {"factor": 8}
    }
}

The option -c/--compress can be used to pass the configuration file.

$ stm32ai analyze -m <model_file> -c <conf_file>.json

Why a small negative ratio is reported for the weights size with a model w/o compression?

X-CUBE-AI optimizer implements different merging engine, allowing for example to fold a batch-normalization layer in the previous layer. In this case the reported value take account of these optimizations, the parameters of the batch-normalization layer are removed in the generated C-model.

$ stm32ai analyze -m ds_cnn.h5
...
weights (ro)       : 159,536 (155.80 KiB) (-0.64%)
...

Is it possible to dump/capture the intermediate values during the execution of the inference?

Yes, thanks the usage of the Platform Observer API (refer to [API], “Platform Observer API” section)

Validation process

How to validate a specific network when multiple networks are resident into the same firmware?

Inside a firmware which is generated with multiple networks, by construction, each generated c-model has its own name. The “validate” command should be used with the argument '--name/-n <c-name>' to indicate the c-model which will be used, else the default name “network” is used.

$ stm32ai validate -m original_model_path --mode stm32 -n net1_name

After the connection phase, the list of the networks which are in the firmware are reported.

...
<Stm32com id=0x28074647320 - CONNECTED(COM35/115200) devid=0x450/STM32H743/753 and STM32H750 msg=2.1>
 0x450/STM32H743/753 and STM32H750 @480MHz/240MHz (FPU is present) lat=4 Core:I$/D$
 found network(s): ['net1_name', 'net2_name',...]
...

Note that after this step, there is a basic checking process to ensure that the selected c-model is compliant with the original model. This “signature” is based on a set of values: TOOLS versions, MACC, RAM and ROM size. Don’t forget to pass also the arguments (compression factor, allocate-inputs…) which were used to generate the c-model else the “signature” will be invalid.

Reported STM32 results are incoherent

Compared to the metrics reported by a validation on desktop, sometimes the STM32 results (validation on target, automatic or not) are different or incoherent. Unfortunately, this can appears because the defined stack or heap sizes are not enough for the generated project.

The work-around is to generate or to modify an aiValidation project, to increase manually the defined stack and/or heap sizes before to re-compile and to flash the STM32 firmware.

GCC-base IDE project

/* Linker file : STM32XXX.ld */
...
_Min_Heap_Size = 0x2000 ;   /* required amount of heap  */
_Min_Stack_Size = 0x800 ;   /* required amount of stack */
...

IAR Embedded Workbench IDE project

Unable to perform automatic validation on-target

Different reasons can interrupt the process of validation.

1 - During the compilation phase of the temporary project

The following typical link issue can appear (IAR Embedded Workbench IDE):

Error[Lp011]: section placement failed
            unable to allocate space for sections/blocks with a  
total estimated minimum size of 0x3'd724 bytes (max align  
0x8) in <[0x2000'0000-0x2001'ffff]> (total uncommitted space  
0x2'0000).

This indicates generally that the input/output tensors and/or activation buffer can be not placed simultaneously in the default or same RW section. A possible work-around is to place the activation buffer in the external memory device (refer to [2]).

Another work-around is to place manually the different buffers in different RAM memory regions.

/* file: Middlewares/ST/Application/Validation/Src/aiValidation.c */
/* file: Middlewares/ST/Application/SystemPerformance/Src/aiSystemPerformance.c */
...
AI_ALIGNED(4)
static ai_u8 activations[AI_NETWORK_DATA_ACTIVATIONS_SIZE];

AI_ALIGNED(4)
static ai_u8 in_data[AI_NETWORK_IN_1_SIZE_BYTES];

AI_ALIGNED(4)
static ai_u8 out_data[AI_NETWORK_OUT_1_SIZE_BYTES];
...

2 - During the set-up of the communication with the board

...
ON-DEVICE STM32 execution ("network", auto-detect, 115200)..
LOAD ERROR: STM32 - no connected board(s), invalid firmware or the board should be re-started

This illustrates the case where a device seems not correctly flashed/re-started or that the board is not connected to the workstation. To check it this point, the best way is to use a terminal application (like TeraTerm or Putty) and to check the init log. COM port and baud-rate information should be used to precise the connection through the UI validation interface (refer to [2]). This test allows to check that the firmware is correctly flashed.

3 - During the communication with the board

After the connection/discovery phase, the following read timeout message can appear after the message Running with inputs=…

...
ON-DEVICE STM32 execution ("network", auto-detect, 115200)..

<Stm32com id=0x22ba711dd30 - CONNECTED(COM35/115200) devid=0x450/STM32H743/753 and STM32H750 msg=2.1>
 0x450/STM32H743/753 and STM32H750 @480MHz/240MHz (FPU is present) lat=4 Core:I$/D$
 found network(s): ['network']
 description    : 'network' uint8,(49, 40, 1)-[5]->uint8,(1, 1, 4) macc=336084 rom=16.30KiB ram=4.25KiB
 tools versions : rt=(4, 1, 0) tool=(4, 1, 0)/(1, 3, 0) api=(1, 1, 0) "Wed Sep 18 22:28:49 2019"

Running with inputs=(10, 49, 40, 1)..
.
LOAD ERROR: STM32 - read timeout 50000ms

or

LOAD ERROR: STM32 - read timeout 10000ms

The reached timeout of 10000ms, indicates that the host has not received a ACK message after the transfer of the input buffer. 50000ms indicates that the STM32 has not sent a end-of-computation message. Both cases indicate generally a STM32 hard-fault. The more common reason is an issue of the sizing of the stack or/and the heap (see previous “Incoherent reported STM32 results” section).

Long time process or crash with a large test data set

With a large data validation/test set, as the data are completely loaded in memory before its usage, according the desktop memory resources, it is possible that the station hangs or crashes after a non-usual time. To avoid this situation, only a representative and limited part of the validation or test data set should be provided. The whole data set is not requested to evaluate the generated model.

Example of Python script to create a “small” test data set (based on a Numpy npz file):

from __future__ import absolute_import, division, print_function, unicode_literals

import numpy as np

# Load the whole data set
arrays = np.load('large_data_set.npz')

# Select the test data set
x_test, y_test = arrays['x_test'], arrays['y_test']

# Select randomly 100 samples (fixed seed)
msize = min(100, len(x_test))
np.random.seed(123)
rchoice = np.random.choice(len(x_test), size=msize, replace=False)
x_test, y_test = x_test[rchoice], y_test[rchoice]

# Save the selected data
np.savez('small_data_set.npz', x_test=x_test, y_test=y_test)

Quantization and post-training quantization process

Backward compatibility with X-CUBE-AI 4.0 and X-CUBE-AI 4.1

X-CUBE-AI 4.0/4.1 reshaped Keras h5 file and associated tensor format configuration file (JSON file) are always supported without adaptation.

The Keras post-training configuration file and associated Python modules can be always used. Only the X-CUBE-AI 4.0 configuration file (JSON file) should be upgraded to add the new mandatory fields: “arithmetic”, “weights_integer_scheme” and “activations_integer_scheme” (refer to [QUANT], “Quantize command” section).

Note — Readers should be aware that the Keras quantization script is fully and only based on the Keras API from the TensorFlow module (version 2.3.1).

Is it possible to use the Keras post-training quantization process through the UI?

No. This feature is only available through the CLI.

Is it possible to use the Keras post-training quantization process with a non-classifier model?

Yes, classification accuracy or loss metrics are not used for the quantization of the weights and activation tensors.

Is it possible to use the compression for a quantized model?

No. The '--compression' option is not supported for the quantized layers. If a compression factor is requested, the following error is generated:

NOT IMPLEMENTED: Quantizing a compressed tensor is not supported for <name_layer>

How to apply the Keras post-training quantization process on a non-Keras model?

This feature is only supported for Keras model (refer to [QUANT])

TensorFlow lite, OPTIMIZE_FOR_SIZE option support

https://www.tensorflow.org/lite/performance/post_training_quantization

Post-quantization TensorFlow lite script (TFLiteConverter, TF 1.15) allows to generate a weight-only quantized file (OPTIMIZE_FOR_SIZE option). This simplest scheme (also called “hybrid” quantization) allows to reduce the size of the generated file (~by 4). Only the weights from floating point to 8-bits of precision are quantized. At inference, weights are converted from 8-bits of precision to floating point and computed using floating-point kernels.

This quantization scheme is not supported by X-CUBE-AI, in particular by the C-inference engine and operator implementations (network runtime library), mainly due to the MCU resource constraints. Additional RAM memory to cache the un-compressed parameters is requested to reduce the latency. If this model is imported, the parameters are converted to floating point before code generation. Only the full 8b integer quantization of weights and activations scheme is supported.
