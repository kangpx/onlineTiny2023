How to know the version of the deep-learning framework components which are used?

X-CUBE-AI Expansion Package is a complete self-contained application package. No external tools is requested to use the package. To know the version of the main components which are embedded, the following command can be used:

$ stm32ai --tools_version
Neural Network Tools for STM32 family v1.7.0 (stm.ai v8.1.0)
- Python version   : 3.9.12
- Numpy version    : 1.23.0
- TF version       : 2.12.0
- TF Keras version : 2.12.0
- ONNX version     : 1.10.2
- ONNX RT version  : 1.15.0
- QKeras version   : 0.9.0
- LARQ version     : 0.12.0

Warning — User should be aware that the respective Python DL modules are used to run the original model and also to import/parse the model. To know the supported layers/operators, please refer to the description in the [TFLITE], [KERAS] and [ONNX].

Channel first support for ONNX model

Data format of the generated C-model are always channel last (or NHWC data format). To conserve the original data arrangement, by default the transpose operators will be added if:

  • model is channel first
  • the number of input channels is greater than 1

This default behavior can be disabled with the option: --no-onnx-io-transpose

Tip — For the validation of the ONNX model, the data should be correctly presented by the user to build the representative data set according the generated and expected input shape.

How is used the CMSIS-NN library?

The network runtime library is partially based on the CMSIS library. CMSIS is a vendor-independent hardware abstraction layer for micro-controllers that are based on Arm® Cortex® processors (https://arm-software.github.io/CMSIS_5/General/html/index.html). The CMSIS-NN sub-component and a minor part of the CMSIS-DSP sub-component are embedded in the network_runtime.a library to be sure that the requested files are compiled with the optimal options.

component version
CMSIS package r5.9.0
CMSIS-Core(M) V5.6.0
CMSIS-DSP V1.10.0
CMSIS-NN V3.1.0

A part of the forward kernel functions are directly mapped on the CMSIS-NN (legacy support of the Qmn format) when available but the major part, are the optimized implementations (integer and float format) only based on the CMSIS-type definitions for portability across the STM32 families.

What is the EABI used for the network_runtime libraries?

For performance motivations, the provided libraries (network_runtime.a) are compiled with the hard floating-point ABI option allowing generation of floating-point instructions and using FPU-specific calling conventions. This implies that the final end-user project should be also compiled and linked with this option.

Warning — Abnormal situation is normally detected at link time, but there is a specific case where the firmware is generated w/o errors but at run-time the results are UNPREDICTABLE. For example for a GCC-base environment, -mfloat-abi=soft or softfp can be set by default for the whole project to use an other EABI-soft-base binary library. There is currently no link issue because the library has been generated with another EABI compliant tool-chain (IAR for ARM tool-chain).

If specific binary library version is requested see with your local ST support.

Is it possible to use the AI stack in a C++ environment?

Yes - The generated 'network.c' and 'network_data.c' files should be compiled with the C compiler. network_runtime static libraries are fully compiled with a C-compiler. For the C++ application code, the AI header files should be included as follow:

#include <iostream>
using namespace std;

extern "C" {
#include "network.h"
#include "network_data.h"
}

X-CUBE-AI Python API availability?

X-CUBE-AI is only available through an UI interface (fully integrated in STM32CubeMX) and through a command line interface. Only a specific Python module (ai_runner) is provided to be able to run locally a generated c-model for advanced validation flow purpose (see “Python-based environment” section in [C-RUN]).

Requested size of the input buffer is not aligned with the shape and data-type of the tensor?

Normaly the size of the buffer ('AI_<NAME>_IN/OUT_SIZE_BYTES') to handle the data of the input tensor is aligned with the shape and the data-type. With a batch dim of 1, the requested size is:

size_in_byte = prod(shape[1:]) * sizeof(data item)

Nevertheless, in some case, extra bytes can be requested to be aligned with the read accesses (4-bytes aligned) performed by the c-kernels. If these read accesses refer a non-memory mapped region, HW exception can be generated. To avoid this situation, the code generator adds if necessary the extra bytes (max. 3 bytes) to the requested “normal” size according to the parameter and the type of first operator.

Note — These extra bytes should be always placed at the end of the buffer and will be accessed in read mode only. Base address of the allocated buffer must be always 4-bytes aligned.

Following C-defines are generated for a quantized model with input RGB image (128x128x3).

/* @file: network.h */

#define AI_NETWORK_IN_1_HEIGHT      (128)
#define AI_NETWORK_IN_1_WIDTH       (128)
#define AI_NETWORK_IN_1_CHANNEL     (3)
#define AI_NETWORK_IN_1_SIZE        (128 * 128 * 3)
#define AI_NETWORK_IN_1_SIZE_BYTES  (49153)    /* = 128x128x3 + 1 */

Stateful LSTM/GRU support?

For the ONNX LSTM and TFlite UNIDIRECTIONAL_SEQUENCE_LSTM operators, only stateless mode is supported. There is only a limited support for the Keras LSTM/GRU layer in stateful mode.

How is used the ONNX optimizer?

The import of the ONNX model is based on the ONNX package 1.6.0. The ONNX optimizer is always used with the standard options to apply the minimum graph optimizations to perform fast inference. The X-CUBE-AI optimization passes are applied after. Some particular models or corner cases are sometimes not correctly supported by this version of ONNX optimizer generating specific internal errors. The --no-onnx-optimizer option can be used to disable the ONNX optimizer pass to import the model.

Example of error message:

(op_type:LeakyRelu, name:LeakyRelu_26): Inferred shape and existing shape differ in rank: (3) vs (2)
INTERNAL ERROR: list index out of range

How is used the TFLite interpreter?

For the built-in validation flow, the TFLite interpreter is used to generate the predictions which will be compared to the outputs of the generated c-model. As the X-CUBE-AI c-runtime inference engine is stateless for the supported operators, between two samples, reset_all_variables() method is called.

How to import a TensorFlow model (floating-point model)?

  • https://www.tensorflow.org/lite/models/convert
  • https://www.tensorflow.org/api_docs/python/tf/lite/TFLiteConverter

Tensorflow model can be not directly imported in X-CUBE-AI. There is two ways to export a TensorFlow model that could be imported:

  • save the TensorFlow model as a Keras model file. In this case, the generated Keras file (h5 file) should be directly imported. This approach is only possible for the model using the high level Keras API.
  • use the TFLiteConverter to create a TFlite file (tflite file). For the model using the high level Keras API, this approach can be also used to apply the built-in optimizations (folding of the BN,..) but also where specific Keras custom layers are used and deployable as the TFLite operators.
import tensorflow as tf

converter = tf.lite.TFLiteConverter.from_saved_model(saved_model_dir)
# converter = tf.lite.TFLiteConverter.from_keras_model(model)
tflite_model = converter.convert()
open("converted_model.tflite", "wb").write(tflite_model)

TensorFlow Keras (tf.keras) vs Keras.io

X-CUBE-AI 7.1 pack embeds only TensorFlow 2.7.0. The Keras.io module is no more used. 'tf.keras 2.7.0' is now used to import and to validate the Keras models. The previous 'TF_KERAS=False' environment variable (up to X-CUBE-AI 5.2) can be no more used to import a Kerias.io back-end. This implies that the Keras model with a NCHW tensor format (channel-first) can be no more validated. Code can be always generated but the validation flow to test the outputs against the outputs of the generated c-model can be no more used. Following type of error message will be generated during the execution of the provided model.

...
INTERNAL ERROR: The Conv2D op currently only supports the NHWC tensor format
                on the CPU. The op was given the format: NCHW [Op:Conv2D]
...

If the user wants to validate the generated c-model, he must provide a test data set. The '--no-exec-model' option can be used to avoid to execute the imported model.

> stm32ai validate -m <keras_model_with_NCHW>.h5 -vi test.npz --no-exec-model
...

It is possible to update a model on the firmware w/o having to do a full firmware update?

Yes - The '--binary' and '--relocatable' options of the 'generate' command allow to implement a simple or complete mechanism to be able to upgrade a whole generated c-model w/o having to do a full firmware update (refer to [CLI] or [RELOC], “Relocatable binary model support” article).

Keras Model or Sequential layer support?

Nested topologies are not supported. This can appear when the Keras functional API is used to build the network and a 'Model' object is called as a layer.

A possible work-around is to convert the Keras model to a TensorFlow lite model, if all operators are supported.

converter = tf.lite.TFLiteConverter.from_keras_model_file(<keras_model_path>)
model = converter.convert()

Is it possible to split the weights buffer?

Yes, by weights/bias tensors, see the '--split-weights' option (refer to [CLI]).

Is it possible to place the “activations” buffer in different memory segments?

Yes, with the X-CUBE-AI 7.1, the user has the possibility to generate a c-model with multiple activations buffers (or heap).

How to compress the non-dense/fully-connected layers?

Only the 32b floating-point dense/fully-connected layer can be compressed.

Is it possible to apply a compression factor different of x8, x4?

No. The underlying weight-sharing algorithm (K-means clustering) is based on a dictionary with 16 (x8) or 256 (x4) entries. At the end, the global gain depends of the parameter number versus dictionary size. Note that for a given layer, bias parameters are not necessarily compressed. Compression is applied by tensor.

How to specify or to indicate a compression factor by layer?

By default, the compression process tries to apply globally the same compression level for all eligible operators. A JSON file can be defined, to indicate what is the compression level which must be applied for a given layer. The layer is specified by its original name (refer to “Specify a compression level by operator” section [CLI])

Why a small negative ratio is reported for the weights size with a model w/o compression?

X-CUBE-AI optimizer implements different merging engine, allowing for example to fold a batch-normalization layer in the previous layer. In this case the reported value take account of these optimizations, the parameters of the batch-normalization layer are removed in the generated C-model.

$ stm32ai analyze -m ds_cnn.h5
...
weights (ro)       : 159,536 (155.80 KiB) (-0.64%)
...

Is it possible to dump/capture the intermediate values during the execution of the inference?

Yes, thanks the usage of the Platform Observer API (refer to [API], “Platform Observer API” section) in C-environment (STM32 or X86) or in Python environment with the ai_runner module (refer to [C-RUN] “How to run locally a c-model” article).
